CC1链逆向分析

漏洞链终点——可执行任意代码的InvokerTransformer

构造反序列化利用链的最终目的是为了执行任意代码，即执行Runtime.getRuntime().exec("calc");

InvokerTransformer类的transform方法可以通过反射实现任意代码执行。

寻找自动触发InvokerTransformer的tranform方法的利用链

TransformedMap满足了这个要求，其checkSetValue调用了valueTransformer的transform方法

而valueTransformer是在decorate这一静态方法调用时传入的Transformer类对象(可控)

那么问题就转化为了构造TransformedMap后如何触发checkSetValue

TransformedMap父类AbstractInputCheckedMapDecorator的setValue调用了parent.checkSetValue

分析代码可知，当调用TransformedMap对象（即被decorate修饰后的map）的键值对entry的setValue方法时，将触发TransformedMap对象的checkSetValue方法。

经过以上的变形，我们把触发条件由原来的invokerTransformer.transform(input)这种显示调用的方式（开发中几乎不会出现），转化为了对map的值进行修改操作（常规操作，在开发中更为常见）。

此时，问题又转化为了寻找用上述遍历entry的方式调用了setValue的类。

寻找反序列化触发点——在readObject中调用了setValue的类

AnnotationInvocationHandler的readObject方法中调用了memberValue.setValue，不仅满足了遍历map并调用setValue方法，而且可作为反序列化漏洞的触发点。

分析其构造函数

memberValues可控，构造是将其赋值为TransformedMap转化后的map即可。该构造函数需要通过反射创建。若能成功利用annotationInvocationHandler，我们就可以实现传入序列化的payload，服务端自动将payload反序列化，并对反序列化后的map进行修改，最终达到调用InvokerTransformer.transform，实现任意代码执行的目的。

然而想要成功执行利用链，还存在以下几个问题：

  • memberValues.setValue方法的参数值不可控，而我们要想顺利执行代码，需要将Object input = Class.forName("java.lang.Runtime").getMethod("getRuntime").invoke(Class.forName("java.lang.Runtime"));传入setValue中
  • Runtime类没有继承serialize接口，Runtime对象不可以序列化。

然而，新的问题出现了

问题出现在input.getClass()这个方法上：

  • 当input是一个类的实例对象时，获取到的是这个类
  • 当input是一个类时，获取到的是java.lang.Class

我们尝试在java.lang.Class中找getRuntime方法，自然就失败了。

那么是否可以直接传递Runtime类的实例呢？

答案是不行的，在writeObject的时候会出现异常： java.io.NotSerializableException: java.lang.Runtime。

因为Runtime类没有实现 java.io.Serializable 接口，不允许被序列化。所以我们只能使用Runtime.class，Class类实现了Serializable接口，可以被序列化。
