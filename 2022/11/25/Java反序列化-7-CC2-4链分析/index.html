Java反序列化(7)-CC2/4链分析

前言

之前的几条利用链都是利用了commons-collections3.2.1的包，commons-collections4作为⼀个新的包，CC1等旧的链是否仍能利用？4.0版本下是否存在新的利用链呢？本文就以上两个问题展开分析。

旧利用链在4.0版本下的微调

以CC1链为例，分析其在4.0版本下的变化。

修改pom.xml，两个包属于不同的分支，两者的命名空间不冲突，可以共存在同⼀个项目中。

在《Java反序列化(3)-使用LazyMap构造CC1链》的代码上进行修改，将所有 import org.apache.commons.collections.* 改成 import org.apache.commons.collections4.*

发现LazyMap中出现报错信息。

原因是4.0版本下除去了decorate方法，将其改成lazyMap方法即可。

寻找4.0版本下的新链

怎样会造成Java反序列化漏洞？

入口类(source点)的参数中包含可控类，并且该可控类在readObect方法中被调用，可作为跳板，经过有限次跳转、调用(套娃)，最终调用某个类的危险方法(sink点)，造成RCE、SSRF危害。

而在commons-collections中寻找利用链，其实就是寻找一条从某个类的readObect方法到Transformer#transform方法的调用链。

TransformingComparator在4.0版本下实现了Serializable接口，其compare方法中调用了transform方法，而compare方法又是开发中较为常见的方法。在CC2和CC4中，找到了java.util.PriorityQueue作为入口点，其readObject方法最终可以调用 TransformingComparator#compare，最终完成利用链的构造。

PriorityQueue的readObject方法中调用了heapify方法。

当size>1时，进入for循环，调用siftDown方法。

siftDown方法又会触发siftDownUsingComparator方法。

该方法中调用了comparator.compare，将comparator赋值为TransformingComparator。

TransformingComparator的compare方法触发transformer.transform。

先尝试修改CC1链的前半部分来实现新的入口。

transformers数组部分不变。

实例化PriorityQueue对象，第⼀个参数是初始化时的大小，第⼆个参数是比较时的Comparator，传入实例化的transformingComparator；前面说到，size需要大于1，所以queue至少有两个元素，元素任意，因为Transformer数组中用到了ConstantTransformer，无需关心传入的元素；其余的部分与CC1相同。

CC4分析

CC2中通过动态字节码执行任意代码，通过TransformingComparator.compare触发InstantiateTransformer.transform。

CC2分析

CC4中没有使用transformer数组构造利用链，而是通过InvokerTransformer#transform直接触发TemplatesImpl#newTransformer。那么与上一篇文章《Java反序列化(6)-Shiro-550》类似，需要想办法将“TemplatesImpl的实例化对象templates”作为参数传给“InvokerTransformer的transform方法“。

通过分析可以发现，transformingComparator#compare的参数obj1，是队列queue里的元素通过PriorityQueue#siftDownUsingComparator的第二个参数x传入的。即队列queue中的元素就是传给 TemplatesImpl#newTransformer 再传递给InvokerTransformer#transform的参数。
