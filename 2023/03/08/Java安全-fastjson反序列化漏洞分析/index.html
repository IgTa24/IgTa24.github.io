Java安全-fastjson反序列化漏洞分析

fastjson介绍

简介

fastjson 是阿里巴巴的开源JSON解析库，它可以解析 JSON 格式的字符串，支持将 Java Bean 序列化为 JSON 字符串，也可以从 JSON 字符串反序列化到 JavaBean。

使用toJSONString方法，可以将对象转换为 json 格式的字符串

但是无法从字符串中获悉原对象所属类

可见，通过添加SerializerFeature.WriteClassName参数，会在生成的字符串中添加键名为 @type 、键值为原对象所属类的键值对。

反序列化

通过JSON.parseObject方法可以实现json字符串的反序列化，而JSON.parseObject通过调用静态方法JSON.parse生成原生对象，再通过(JSONObject) JSON.toJSON(obj);返回 JSONObject 对象（JSONObject 是一种map）。

做一些测试。

从结果中大致可以推断，JSON.parseObject方法在反序列化字符串时，触发JSON.parse方法，该方法会根据 @type 字段调用类的以 set 开头的方法还原对象，然后触发(JSONObject) JSON.toJSON(obj)转化为JSONObject对象时，会调用类的以 get 开头的方法。

JSON.parseObject方法可以通过指定第二个参数还原对象，但一般情况下该参数都是写死在代码中的，所以漏洞的触发点一般都是JSON.parseObject的第一个参数可控。

以下是一个简单的漏洞例子，很像一句话木马。

具体的漏洞触发成因需要通过调试源代码分析。

漏洞分析

由于JSON.parseObject方法的调用流程很复杂，本部分分析只挑选重点代码部分。

JSON.parseObject(s); -> Object obj = parse(text); -> parse(text, DEFAULT_PARSER_FEATURE); -> DefaultJSONParser#parse -> DefaultJSONParser#parseObject

根据@type值，调用loadClass加载了恶意类。

继续调试，DefaultJSONParser#parseObject -> ParserConfig#getDeserializer -> ParserConfig#createJavaBeanDeserializer -> JavaBeanInfo#build

通过@type获取类之后，利用反射拿到了该类所有的方法并存入 methods，然后在JavaBeanInfo#build方法中遍历 methods 获取getter、setter 方法。

set的查找方式：

  1. 方法名长度大于4
  2. 非静态方法
  3. 返回值为void或当前类
  4. 方法名以set开头
  5. 参数个数为1

获取 setter 方法的其中一个目的是为了还原类的属性，且该方法会对属性做一些默认处理，如转为小写、去除开头的_等，如由 setCmd 方法获得的属性为 cmd。

第二个 for 循环获取类的所有公有静态属性

再下一个 for 循环则是

get 的查找方式：

  1. 方法名长度大于等于4
  2. 非静态方法
  3. 以get开头且第4个字母为大写
  4. 无传入参数
  5. 返回值类型继承自 Collection、Map、AtomicBoolean、AtomicInteger、AtomicLong 中的一个
  6. setter 方法中没有的属性

继续调试，JavaBeanInfo#build -> JavaBeanInfo#JavaBeanInfo -> ParserConfig#createJavaBeanDeserializer

在ParserConfig#createJavaBeanDeserializer方法中，若 asmEnable 为 true，则会调用asmFactory.createJavaBeanDeserializer创建 JavaBean 反序列化器，无法调试。为了便于调试，我们可以尝试令fieldInfo.getOnly为 true，从而将 asmEnable 赋值为 false，通过 new JavaBeanDeserializer(this, clazz, type)创建的反序列化器就可以进行调试了。

搜索对 getOnly 赋值的地方，如图，FieldInfo 的构造函数中（之前的三个for循环中都调用了该构造函数），当满足一定条件时将把 getOnly 赋值为 true。条件为方法的参数个数不为1，而setter方法又要求参数个数为1，getter方法也需要满足其本身的查找要求才会调用FieldInfo 的构造函数。

在 Evil 类中添加一个没有 setter 方法的 getter 方法（getOnly顾名思义），且满足 getter 方法的查找条件。

ParserConfig#createJavaBeanDeserializer -> JavaBeanDeserializer#JavaBeanDeserializer -> JavaBeanDeserializer#deserialze -> JavaBeanDeserializer#parseField -> DefaultFieldDeserializer#parseField -> FieldDeserializer#setValue

该方法中触发了method.invoke(object, value)，存在安全风险。

小结

  • fastjson 在反序列化 json 字符串时，会根据指定的@type类型创建反序列化器进行反序列化，在反序列化过程中，会调用满足条件的 setter 方法和 get 方法。
  • fastjson反序列化与之前几篇文章所写的反序列化不同，source 点并不是 readObject，但 sink 点是相同的，即能执行恶意字节码的点或或能通过反射执行命令的点。
  • fastjson 的反序列化特性，使得当 json 字符串可控时，可以自动触发某个类的 setter 和 getter 方法，从而形成反序列化链。

BCEL 反序列化

Apache Commons BCEL 是 Apache Commons 项目下的一个子项目，被包含在了 JDK 的原生库中。

Repository 用于将一个 Java Class 转换成原生字节码，Utility 用于将原生的字节码转换成 BCEL 格式的字节码

com.sun.org.apache.bcel.internal.util.ClassLoader#loadClass中调用了defineClass(class_name, bytes, 0, bytes.length)可以用以触发恶意字节码。

类名需要以 $$BCEL$$ 开头，又因为 createClass 中对类名进行了 Utility.decode，所以需要用 Utility.encode 处理类名。

如何自动触发 loadClass？需要借助 tomcat 包下的 BasicDataSource。

漏洞修复与绕过

fastjson-1.2.25

在版本 1.2.25 中，官方对之前的反序列化漏洞进行了修复，引入了 checkAutoType 安全机制，默认情况下 autoTypeSupport 关闭，不能直接反序列化任意类，而打开 AutoType 之后，是基于内置黑名单来实现安全的，fastjson 也提供了添加黑名单的接口。

影响版本：1.2.25 <= fastjson <= 1.2.41 描述：作者通过为危险功能添加开关，并提供黑白名单两种方式进行安全防护，其实已经是相当完整的防护思路，而且作者已经意识到黑名单类将会无穷无尽，仅仅通过维护列表来防止反序列化漏洞并非最好的办法。而且靠用户自己来关注安全信息去维护也不现实。

payload：

fastjson-1.2.42

在版本 1.2.42 中，fastjson 继续延续了黑白名单的检测模式，但是将黑名单类从白名单修改为使用 HASH 的方式进行对比，这是为了防止安全研究人员根据黑名单中的类进行反向研究，用来对未更新的历史版本进行攻击。同时，作者对之前版本一直存在的使用类描述符绕过黑名单校验的问题尝试进行了修复。

影响版本：1.2.25 <= fastjson <= 1.2.42

影响版本：1.2.25 <= fastjson <= 1.2.43 描述：上有政策，下有对策。在 L、; 被进行了限制后，安全研究人员将目光转向了 [。
