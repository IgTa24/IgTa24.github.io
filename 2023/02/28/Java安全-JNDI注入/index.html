Java安全-JNDI注入

JNDI简介

Java 命名和目录接口 (Java Naming and Directory Interface) 是一种应用程序编程接口 (API)，它为使用 Java 编程语言编写的应用程序提供命名和目录功能。它被定义为独立于任何特定的目录服务实现。因此，可以以通用方式访问各种目录——新的、正在出现的和已经部署的目录。

Lesson: Overview of JNDI (The Java™ Tutorials > Java Naming and Directory Interface) (oracle.com)

SPI 全称为 Service Provider Interface，即服务供应接口，主要作用是为底层的具体目录服务提供统一接口，从而实现目录服务的可插拔式安装。

JNDI 包含在 Java SE 平台中。要使用 JNDI，必须具有 JNDI类 和一个或多个服务提供程序。JDK 包括以下命名/目录服务的服务提供商：

  • RMI: Java Remote Method Invocation，Java 远程方法调用；
  • LDAP: 轻量级目录访问协议；
  • CORBA: Common Object Request Broker Architecture，通用对象请求代理架构，用于 COS 名称服务(Common Object Services)；
  • DNS: 域名服务

除此之外，用户还可以在 Java 官网下载其他目录服务实现。由于 SPI 的统一接口，厂商也可以提供自己的私有目录服务实现，用户可无需重复修改代码。

JDNI+RMI 攻击

概述

在JNDI服务中，服务端除了直接绑定远程对象之外，还可以通过Reference类来绑定一个外部的远程对象（在命名/目录系统之外）。

Reference 中几个比较关键的属性：

  1. className - 远程加载时所使用的类名
  2. factory - 加载的 class 中需要实例化类的名称
  3. factoryLocation - 提供 classes 数据的地址可以是 file/ftp/http 等协议

小结

当JNDI服务端绑定了一个引用Reference，且Reference的参数factoryLocation可控时，就有可能造成JNDI客户端发起请求后，加载了factoryLocation指向的恶意地址，导致加载恶意字节码从而被攻击。

如果JNDI客户端的lookup方法的参数可控时，也有可能被攻击者事先准备的RMI服务导向恶意字节码所在的地址，导致包含在里面的恶意代码被执行。

回到流程，获得 Reference 后，调用NamingManager#getObjectInstance

该方法主要功能是从 Reference 里生成真正要调用的对象。具体操作在NamingManager#getObjectFactoryFromReference中。

getObjectFactoryFromReference 首先用本地的 AppClassLoader 加载 factoryName 类，找不到就用 URLClassLoader 去 codeBase 的路径中加载，最后实例化。

重点看helper.loadClass，第一个loadClass是对后两个loadClass的封装。第二个loadClass中调用了Class.forName，若被调用则会导致类的初始化（静态代码块被执行）。此处因为通过URLClassLoader加载类，则会进入第三个loadClass，然后再调用第二个loadClass，导致类初始化。

回到NamingManager#getObjectInstance，由于getObjectFactoryFromReference无法将恶意类Evil转化为ObjectFactory，所以此处会抛出异常，可以让恶意类继承ObjectFactory就可以正常返回了。

修复

Java 在 JDK 8u121之后进行了修复，在 RegistryContext 中添加了 默认为 false 的 trustURLCodebase，默认不信任远程代码，也就无法远程加载恶意字节码了

jdk8u/jdk8u/jdk: 557d133dbc61 (openjdk.org)

运行时需要加入参数 -Dcom.sun.jndi.rmi.object.trustURLCodebase=true

否则抛出异常

JNDI+LDAP 攻击

通过流程分析，我们可以知道，JNDI根据不同的协议，调用不同的类的lookup（如rmi协议调用的是RegistryContext），而执行恶意代码的操作是在NamingManager#getObjectFactoryFromReference中进行的，在 JDK 8u121 版本中，还对 CORBA 采取了与 RMI 相似的修复（通过参数 trustURLCodebase 限制远程代码执行），但遗漏了对LDAP的修复。

成功弹出计算器，由下图可以看见利用流程: 受害端发出 ldap 请求，Yakit 返回一个 reference 类，指向一个 web 地址 -> 受害端访问 web 地址，Yakit 返回一个恶意类 -> 受害端加载恶意类导致命令执行。

该漏洞在 8u191 之后才被修复。在 JDK 8u191 之后的版本中，在加载远程对象的 loadClass 方法中加入了 trustURLCodebase 的判断，当 trustURLCodebase 为 true 时才可以进行远程类加载。
